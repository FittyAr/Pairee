# Pairee Plugin Registry — Specification

> **Status: Planned. This document describes the future versioned community plugin registry hosted on the `plugin-registry` Git branch of the Pairee repository.**

---

## 1. Overview

The Pairee Plugin Registry is a community-curated catalog of Lua plugins distributed through a dedicated, standalone Git branch (`plugin-registry`) in the main `FittyAr/Pairee` repository. It provides:

- A versioned, directory-based history of every published plugin release.
- Individual file-by-file SHA-256 integrity verification.
- A structured Pull Request workflow for community submissions.
- A `pairee plugin` CLI for search, list, install, update, check-updates, pin, and removal operations.
- Dynamic keybinding resolver merging, mapping keys defined in plugin manifests directly at load time.
- A secure Developer Mode with format, validate, hash-generation commands, and automated Git branch/PR UI.

---

## 2. Registry Branch Layout

The `plugin-registry` branch is a **standalone orphan branch** (no shared history with `main`). It holds only registry catalog indices and raw plugin directories:

```text
plugin-registry/
├── registry/
│   ├── index.toml                      # Master catalog (auto-generated by CI)
│   └── plugins/
│       ├── git.pairee/
│       │   ├── manifest.toml           # Public plugin manifest
│       │   ├── 1.0.0/                  # Version folder
│       │   │   ├── main.lua
│       │   │   ├── manifest.toml
│       │   │   ├── sha256.sum          # Checksum values for 1.0.0 files
│       │   │   ├── help/
│       │   │   │   └── en.md
│       │   │   └── lang/
│       │   │       └── en.toml
│       │   └── 1.1.0/
│       │       ├── main.lua
│       │       ├── manifest.toml
│       │       ├── utils.lua
│       │       ├── sha256.sum          # Checksum values for 1.1.0 files
│       │       ├── help/
│       │       │   ├── en.md           # Default help documentation
│       │       │   └── es.md           # Localized help in Spanish
│       │       └── lang/
│       │           ├── en.toml
│       │           └── es.toml
│       └── fzf.pairee/
│           └── ...
├── CONTRIBUTING.md                     # How to submit a new plugin
└── VALIDATION.md                       # Schema rules and CI requirements
```

**Branch rules:**
- The branch is protected; only CI and maintainers may push to it.
- Versions are **never deleted** — the registry is append-only.
- Developers submit plugins via Pull Requests containing their new version directories and manifest modifications.

## 3. Schema Definitions

### 3.1 `registry/index.toml`

The index file acts as the master catalog holding metadata for plugin discovery. It does NOT contain file mappings or hashes to minimize size.

```toml
[plugins.git_pairee]
name        = "git.pairee"
version     = "1.1.0"
description = "Git status integration for file panels"
author      = "contributor-handle"
languages   = ["en", "es"]
hooks       = ["on_cd"]
min_pairee  = "0.7.0"
```

### 3.2 Plugin `manifest.toml`

Located inside the plugin workspace directory and at `registry/plugins/<initial>/<author>/<name>/manifest.toml`. During packaging, the file list with SHA-256 hashes is automatically appended to the end of the registry's copy under the `[files]` table:

```toml
name         = "git.pairee"
version      = "1.1.0"
description  = "Git status integration for file panels"
author       = "contributor-handle"
license      = "MIT"
homepage     = "https://github.com/contributor/git.pairee"
tags         = ["git", "vcs"]

# Language integration support
default_language = "en"
languages    = ["en", "es"]

# Settings and options configuration schema
[settings_schema]
show_hidden = { type = "bool", default = false, description = "Show hidden VCS files" }
git_path    = { type = "string", default = "git", description = "Custom Git executable path" }
max_depth   = { type = "integer", default = 3, description = "Max directory recursion depth" }

# Minimum Pairee version required to run this plugin
min_pairee   = "0.7.0"

# Must the user opt-in to trusted mode for this plugin to work?
# Set to true if the plugin uses pairee.fs.spawn or needs os/io access
requires_trust = false

# Declared event subscriptions (informational, also used by CI)
hooks        = ["on_cd"]

# Plugin functional category
# Options: "previewer" | "preloader" | "hook" | "command" | "mixed"
type         = "hook"

# Default Keyboard Shortcuts
[keybindings]
"ctrl+g" = "entry"          # Overlays Ctrl+G to run the plugin's entry() function

# Appended automatically by the packaging tool inside the registry:
[files]
"main.lua" = "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"manifest.toml" = "fa25d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"utils.lua" = "c081d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"help/en.md" = "1a81d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"help/es.md" = "2a81d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"lang/en.toml" = "d081d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"lang/es.toml" = "f081d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
"LICENSE" = "776289021cde9f6d1160160dd8603063af5f701d627e3b74c5f676a93b54aa09"
```

Contains the lowercase hex SHA-256 hash of each file in the version folder:

```

1a81d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855  help/en.md
```

## 4. `plugins.lock` — User-Side Lock File

Stored at `~/.config/pairee/plugins.lock` (Linux/macOS) or `%APPDATA%\pairee\config\plugins.lock` (Windows).

```toml
# Managed by Pairee — do not edit manually.
# Commit this file to version-control your plugin configuration.

lock_version = "1"

[[plugin]]
name    = "git.pairee"
version = "1.1.0"
source  = "registry"

[[plugin.files]]
path   = "main.lua"
sha256 = "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"

[[plugin.files]]
path   = "manifest.toml"
sha256 = "fa25d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"

[[plugin.files]]
path   = "utils.lua"
sha256 = "c081d4a298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
```

## 5. CLI Commands Reference

All plugin management commands run through `pairee plugin <subcommand>`.

| Command | Description |
|---------|-------------|
| `pairee plugin list` | List all installed plugins, their versions, trusted status, and if updates are available |
| `pairee plugin search <query>` | Search the locally cached registry index (`index.toml`) by name, author, or tags |
| `pairee plugin info <name>` | Show metadata, version history, trust requirements, and declared keybindings |
| `pairee plugin install <name>` | Download, verify SHA-256 for each file, and write/install the latest version |
| `pairee plugin install <name>@<ver>` | Install a specific version |
| `pairee plugin check-updates` | Contacts the remote registry branch, compares installed versions, and lists available updates |
| `pairee plugin update` | Update all installed registry plugins to their latest compatible SemVer version |
| `pairee plugin remove <name>` | Uninstall the plugin directory and clean up entries from `plugins.lock` |
| `pairee plugin pin <name>@<ver>` | Pin a plugin to a specific version in `plugins.lock` |
| `pairee plugin verify` | Re-verify SHA-256 of all installed files against `plugins.lock` |

### 5.1 Visual Badges and Tags in Search/List Results

When listing or searching plugins via the CLI (`pairee plugin search` / `pairee plugin list`) or the TUI, the interface highlights:
1. **Type/Category Badge:** Colored labels indicating plugin roles:
   * `[Hook]` (Green) - Event-driven hooks.
   * `[Previewer]` (Cyan) - Custom file panel previewers.
   * `[Command]` (Yellow) - Custom commands/utilities.
   * `[Mixed]` (Magenta) - Plugins implementing multiple functions.
2. **Languages Badge:** A list of supported language locales, showing at a glance if translations exist:
   * `[EN]` `[ES]` `[FR]` (Dim white/gray text).
3. **Tags:** Hash-prefixed descriptive topics.

**Example CLI Search Output:**
```bash
$ pairee plugin search git
git.pairee  [Hook] [EN] [ES]  Git status integration for file panels  #git #vcs
```

## 6. Download & Verification Flow

```
pairee plugin install git.pairee
  │
  ├─ 1. Fetch registry/index.toml via HTTPS
  │       URL: https://raw.githubusercontent.com/FittyAr/Pairee/plugin-registry/registry/index.toml
  │
  ├─ 2. Locate plugin entry in index → resolve latest version and its file mappings list
  │
  ├─ 3. Fetch manifest.toml for the target version
  │       URL: .../registry/plugins/git.pairee/manifest.toml
  │
  ├─ 4. Check min_pairee ≤ current Pairee version
  │       └─ FAIL → print error, abort
  │
  ├─ 5. Download each file individually based on resolved mapping paths
  │       For each file:
  │         ├─ Download file content (e.g. main.lua)
  │         ├─ Compute SHA-256 hash of downloaded file
  │         ├─ Compare against declared hash in index.toml
  │         │    ├─ MATCH → write to ~/.config/pairee/plugins/git.pairee/<path>
  │         │    └─ MISMATCH → delete downloaded assets, print error, abort
  │
  ├─ 6. Append or update entry in plugins.lock
  │
  └─ 7. Print success summary
          ├─ If requires_trust = true → print trust opt-in reminder
          └─ Merge keybinding overlays into active resolver config
```

## 7. Versioning Rules

Plugins follow **Semantic Versioning 2.0.0** (`MAJOR.MINOR.PATCH`):

| Component | When to increment |
|-----------|-------------------|
| `MAJOR` | Breaking change to the plugin's publicly observable behavior |
| `MINOR` | New capabilities, backward-compatible |
| `PATCH` | Bug fixes, no behavioral change |

## 8. Community Plugin Submission Workflow

When `developer_mode = true` is set in `pairee.toml`:
1. The developer prepares their plugin directory locally (`~/.config/pairee/plugins/my-plugin`).
2. Runs validation and signs the files locally:
   ```bash
   pairee developer package ~/.config/pairee/plugins/my-plugin
   ```
   This generates `sha256.sum` containing hashes of every file in the directory.
3. Accesses the **PR Submission TUI** inside Pairee.
4. Inputs GitHub personal access token (PAT), developer fork URL, and commit details.
5. Chooses "Submit Pull Request". The Rust engine:
   * Copies the local directory assets into the correct folder structure under `registry/plugins/<name>/<version>/` in the developer's cloned registry workspace.
   * Auto-commits changes to a new Git branch.
   * Pushes the branch to the developer's GitHub fork.
   * Dispatches a GitHub API request to create the Pull Request targeting the official `plugin-registry` branch.

## 9. Validation Script (`scripts/validate-plugin.sh`)

This script is provided in the repository for local pre-submission validation and is run by the CI system. It performs:

1. **Manifest Schema & Fields Verification:**
   * Checks that `manifest.toml` exists and is valid TOML.
   * Checks required fields: `name`, `version`, `description`, `author`, `license`, `type`, `min_pairee`.
   * Checks `type` is one of: `previewer`, `preloader`, `hook`, `command`, `mixed`.
2. **Directory & File Structure Verification:**
   * Verifies that the version directory exists under the plugin folder and contains `main.lua` and `sha256.sum`.
   * Iterates each file listed in `sha256.sum`, verifies that it exists in the folder, and computes the SHA-256 hash to confirm it matches exactly.
   * Runs `luac -p` to validate Lua syntax of all Lua files.
3. **Cross-Platform Naming Checks:**
   * Verifies that all directory and file names within the package contain ONLY lowercase alphanumeric characters, dots (`.`), dashes (`-`), and underscores (`_`).
   * Rejects names containing spaces, capital letters, or special characters (e.g. `?`, `*`, `:`, `\`, `/`, `|`, `<`, `>`, `"`) which are incompatible across filesystems.
4. **Name Consistency Check:**
   * Verifies that the plugin folder name matches exactly the `name` field declared in `manifest.toml`.
5. **Localization & Help File Coverage:**
   * Compares the `languages` array in `manifest.toml` with the files in `lang/` and `help/` subdirectories.
   * Rejects the plugin if a declared language lacks a corresponding `lang/<locale>.toml` or `help/<locale>.md` file.
   * Rejects the plugin if there are untracked language/help files present in `lang/` or `help/` that are not registered in the `languages` array.
   * Confirms that the `default_language` has both `lang/<default_language>.toml` and `help/<default_language>.md` files.
6. **Translation Key Sync Validation:**
   * Parses all `lang/*.toml` files and extracts the full tree of keys.
   * Raises a warning if there are missing translation keys in any of the localized files compared to the default language or other languages.
7. **Encoding Validation:**
   * Ensures all text files (`.toml`, `.md`, `.lua`) are encoded in valid, well-formed UTF-8.

## 10. CI Workflow (`plugin-registry-ci.yml`)

A GitHub Actions workflow runs on every PR targeting the `plugin-registry` branch:

```yaml
# .github/workflows/plugin-registry-ci.yml
name: Plugin Registry CI
on:
  pull_request:
    branches: [plugin-registry]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate plugin manifests & file hashes
        run: ./scripts/validate-plugin.sh
      - name: Lint Lua files
        run: |
          find registry/plugins -name "*.lua" -exec luac -p {} \;

  regenerate-index:
    if: github.event_name == 'push' && github.ref == 'refs/heads/plugin-registry'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Regenerate index.toml with new file mappings
        run: ./scripts/generate-index.sh
      - name: Commit updated index
        run: |
          git config user.name "Pairee CI"
          git config user.email "ci@pairee"
          git add registry/index.toml
          git commit -m "chore: regenerate plugin index" || echo "No changes"
          git push
```
